# 架构
宗旨：框架实现基础机制，具体功能由函数类自己实现。函数之间松耦合。

## 源码解析
首先进行源码解析 [CanonicalParser](../src/parser/canonicalParser.ts)，得到函数级别的含义。这一步的主要作用是去糖、固化参数。

之前还有一个选择是：先解析成token得到源码级别的层级，然后再赋予函数级别的含义。比如先将源码解析为：函数调用、大括号、标签、未知文本，然后将函数节点转换为未知文本进行去糖。其实是等价的，将去糖和解析一步合并更快。

解析器只负责解析基本语法和调度，函数提供参数定义，解析器根据定义把解析好的内容交给函数，函数自行处理参数。如果遇到基础语法之外的字符，则尝试用语法糖解析，此时解析器调用所有函数的去糖函数。如果没有一个去糖函数能消费，则认为是无用字符。

现在的设计是解析器尽职尽责，把饭送到函数嘴边（已经获取了参数的具体类型，用具体参数创建函数）。还有一种设计是去解析器化，即函数自己处理字符串，相当于将解析函数融入节点类中，比如说函数节点构造的时候传递的都是字符串信息。和现在的区别在于，现在相当于先获取子节点，再创建父节点；而该方案相当于先创建父节点，再创建子节点，或者说子节点的创建包含于父节点的创建之中。

简单说一下另一个设计的具体做法：所有解析函数写在基类中，具体函数节点是基类的子类，每个节点的构造函数：
```ts
constructor(args: Map<number|string, SourceSpan>, ctx: ParseContext, span: SourceSpan) {
    this.content = super.parseContent(args.get('content') ?? args.get(0), ctx);
    this.n = super.parseNumber(args.get('n') ?? args.get(1), ctx);   // 这个函数内部会完成报错等操作
}
```

目前遇到的难点在于，解析结束后的wrapup，这是语法糖带来的需求，比如最后剩下的几个note需要用bar包裹，最后的几个bar需要用voice包裹等等。具体包裹到什么程度需要由父节点的层级决定，比如父节点是bar，则需要子节点进行bar以下层级的wrapup。

这个wrapup必须从最底层往上，而不能仅仅进行低一级的wrapup。例子：假设有这样的层级关系：
```
note -> bar -> 声部Voice -> 一行Line -> 曲谱Score
```
对于Score层级，依赖Line，假设只有note，需要先变为bar再变为voice最后归为Top。现解析完剩下一堆note，如果仅仅对低一层的Line收束会发现无法直接消费note。因此对于Score，需要先完成bar的wrapup，再Voice的wrapup，最后Line的wrapup，最后组装Lines为Score。

有的时候并非所有content都需要wrapup。因此是否wrapup取决于函数自己。比如 Voice(一堆note或bar) 就需要wrap成很多bar，但是临时多声部 over(一堆note或bar, 一堆note或bar) 就不需要。

如果要在现在的架构里实现，就需要在函数内部进行，因为parser不管函数具体逻辑。这样就耦合在一起了。如果是上面的另一种设计，AST节点就和解析是一个类。
